# Техническое задание (ТЗ)
## на разработку консольного приложения для генерации фотоотчетов в MS Word

**Версия документа:** 1.0  
**Дата составления:** 24.05.2024  
**Язык реализации:** Python 3.12+

---

## 1. Общие сведения
### 1.1. Наименование проекта
Консольное приложение `PhotoReportGenerator` для автоматизированного создания структурированных фотоотчетов в формате `.docx`.

### 1.2. Назначение системы
Приложение предназначено для автоматической обработки директории с изображениями, их классификации по ориентации и формирования готового документа MS Word, где фотографии размещены сеткой на страницах формата А4 с максимальным использованием полезной площади листа.

### 1.3. Целевая аудитория
Пользователи, не обладающие глубокими техническими знаниями, нуждающиеся в быстрой подготовке печатных фотоотчетов (например, для архивов, отчетов по командировкам, портфолио).

---

## 2. Требования к окружению и зависимостям
### 2.1. Программная платформа
*   **Язык программирования:** Python версии 3.12 или 3.13.
*   **Операционная система:** Кроссплатформенность (Windows 10/11, macOS, Linux).
*   **Режим запуска:** Командная строка (CLI / Terminal).

### 2.2. Внешние библиотеки
Проект должен использовать следующие сторонние библиотеки (указываются в `requirements.txt`):
*   `python-docx` — для программной генерации и форматирования документов MS Word.
*   `Pillow` (PIL) — для анализа метаданных изображений (размеры, ориентация) без обязательной загрузки всего изображения в память.
*   `argparse` — стандартная библиотека для обработки аргументов командной строки.
*   `tqdm` (опционально) — для отображения прогресс-бара обработки файлов.

### 2.3. Портативность
Приложение не требует установки полноценной IDE. Запуск осуществляется через интерпретатор Python. Допускается возможность последующей компиляции в `.exe` (через PyInstaller), но в рамках данного ТЗ приоритетом является работоспособность скрипта.

## 3. Функциональные требования
### 3.1. Входные данные
*   Пользователь указывает путь к локальной папке, содержащей изображения.
*   Пользователь указывает желаемое количество фотографий на одной странице (целое число > 0).
*   Поддерживаемые форматы изображений: `.jpg`, `.jpeg`, `.png`, `.webp`, `.bmp`.
*   Регистр расширений файлов не должен влиять на распознавание (`.JPG` и `.jpg` равнозначны).

### 3.2. Обработка изображений
*   **Сканирование:** Рекурсивный или линейный обход указанной папки.
*   **Валидация:** Проверка файла на корректность (возможность открытия библиотекой Pillow). Поврежденные файлы должны пропускается с записью предупреждения в лог, но не должны прерывать работу программы.
*   **Классификация:** Автоматическое определение ориентации на основе соотношения сторон (Width/Height):
    *   **Альбомная (Landscape):** Ширина больше Высоты.
    *   **Книжная (Portrait):** Высота больше или равна Ширине.
*   **Группировка:** Строгое разделение потоков. На одной странице документа **запрещено** смешивание альбомных и книжных ориентаций. Если текущая страница заполняется альбомными фото, а следующее фото книжное — начинается новая страница (или раздел).

### 3.3. Генерация документа (.docx)
*   **Формат страницы:** А4 (210 x 297 мм).
*   **Поля страницы:** Минимально допустимые для печати (рекомендуется 1.0 см со всех сторон) для максимизации места под контент.
*   **Размещение (Layout):**
    *   Использовать таблицу невидимого типа (границы ячеек отсутствуют) для обеспечения сетки.
    *   **1 фото на лист:** Изображение масштабируется пропорционально, занимая максимальную высоту или ширину доступной области (с учетом полей).
    *   **N фото на лист (N > 1):** Изображения распределяются по ячейкам таблицы равномерно.
        *   Пример для 4 фото: сетка 2x2.
        *   Пример для 2 фото: сетка 1x2 или 2x1 в зависимости от ориентации.
        *   Пример для 3 фото: оптимальная сетка (например, 2 сверху, 1 снизу по центру, либо 2x2 с одной пустой ячейкой).
    *   **Отступы:** Между изображениями должен присутствовать небольшой фиксированный отступ (padding) для визуального разделения (эстетика).
*   **Масштабирование:** Изображения не должны искажаться (сохранение_aspect_ratio). Они вписываются в ячейку методом `contain`.
*   **Нумерация:** Автоматическая простановка номеров страниц в нижнем колонтитуле (Footer) по центру или справа.
*   **Имя файла:** Генерируется автоматически по шаблону `PhotoReport_YYYY-MM-DD_HH-MM.docx`. Сохраняется в корневой директории скрипта или в папке источника (настраиваемое поведение).

### 3.4. Интерфейс (CLI)
*   Поддержка аргументов командной строки:
    *   `--path` / `-p`: Путь к папке с фото.
    *   `--count` / `-c`: Количество фото на страницу.
    *   `--output` / `-o`: Путь для сохранения файла (опционально).
*   **Интерактивный режим:** Если аргументы не переданы, программа должна в запросить данные у пользователя через консоль (prompt).
*   **Логирование:** Вывод в консоль статуса обработки (например: "Найдено 50 фото", "Обработка альбомных...", "Сохранено...").

## 4. Требования к надежности и обработке ошибок
### 4.1. Устойчивость к сбоям
*   Отсутствие файлов в папке: Программа должна сообщить об ошибке и корректно завершить работу (код выхода != 0), не создавая пустой документ.
*   Отсутствие прав на запись: При невозможности сохранить файл программа должна вывести понятное сообщение об ошибке доступа.
*   Поврежденные изображения: Файл пропускается, счетчик обработанных файлов уменьшается, в консоль выводится предупреждение.

### 4.2. Производительность
*   Обработка больших папок (500+ фото) не должна приводить к переполнению оперативной памяти (изображения не должны загружаться в RAM полностью, только потоковая обработка или чтение заголовков).
*   Время генерации документа должно быть линейно зависимым от количества файлов.

## 6. Критерии приемки (Testing Criteria)
Приложение считается принятым, если успешно пройдены следующие тесты:
1.  **Тест "Счастливый путь":** Папка содержит 10 альбомных и 10 книжных фото. Программа создает документ, где альбомные на одних страницах, книжные на других. Нумерация страниц сквозная.
2.  **Тест "Одно фото":** При указании `count=1`, каждое фото занимает почти весь лист А4.
3.  **Тест "Сетка":** При указании `count=4`, на листе ровно 4 фото одинакового размера сеткой 2x2.
4.  **Тест "Ошибка пути":** При вводе несуществующего пути программа не падает с traceback, а выводит сообщение "Папка не найдена".
5.  **Тест "Пустая папка":** При отсутствии фото программа сообщает об этом и не создает файл.
6.  **Тест "Битый файл":** В папку добавлен файл с расширением .jpg, но не являющийся изображением. Программа пропускает его и продолжает работу.

## 7. Ограничения
*   Приложение не имеет графического интерфейса (GUI).
*   Приложение не редактирует исходные файлы фотографий (не меняет их размер на диске, только при вставке в Word).
*   Приложение не поддерживает создание PDF напрямую (только через конвертацию из Word пользователем).
*   Приложение не поддерживает облачные хранилища (работает только с локальной файловой системой).
